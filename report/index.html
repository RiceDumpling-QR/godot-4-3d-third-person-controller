RoboBlast · Godot 4.6

Restoring third-person locomotion

We removed the complete locomotion system from a working third-person shooter while leaving its camera, combat, enemies, physics, HUD, and level intact. The task asks a coding agent to reconstruct the missing player experience.

Before and after ablation

Why locomotion?

Locomotion is easy to understand as a player, but recovering it well requires more than reconnecting WASD.

  1. It has clear player impact. The ablated game still looks complete, yet immediately feels broken. Successful recovery is visible without reading code.
  2. It exercises several connected systems. A full solution must coordinate camera-relative input, diagonal speed, acceleration, collision, jumping, facing, animation, footsteps, and landing feedback.
  3. It supports objective partial credit. A headless verifier measures movement, velocity, direction, rotation, jump trajectory, and landing state. Basic recovery can score without being mistaken for a polished solution.
Current scope. This draft documents the task design and behavioral difference. The three agent attempts, difficulty calibration, anti-cheat evidence, and failure analysis will be added after the runs are complete.
